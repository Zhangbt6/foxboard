# FoxBoard 验收链模板

> 版本：v1.0  
> 适用范围：所有 FoxBoard 项目任务  
> 流程：白狐设计 → 青狐实现 → 黑狐审核 → 花火确认

---

## 一、验收流程图

```
┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐
│   🦊 白狐设计    │────▶│   🦊 青狐实现    │────▶│   🦊 黑狐审核    │────▶│   🔥 花火确认   │
│   (设计/调研)    │     │   (开发/执行)    │     │   (质量把关)     │     │   (最终验收)    │
└─────────────────┘     └─────────────────┘     └─────────────────┘     └─────────────────┘
        │                       │                       │                       │
        ▼                       ▼                       ▼                       ▼
   设计文档/原型           代码/实现产物            审核报告               归档确认
   验收标准清单            单元测试结果            质量评级               下一Phase规划
```

## 二、各环节验收标准清单

### 2.1 白狐设计阶段 🦊

| 检查项 | 验收标准 | 通过标准 |
|--------|----------|----------|
| 需求理解 | 是否完全理解花火派单需求 | 能用1句话复述任务目标 |
| 设计产出 | 是否输出设计文档/原型/调研报告 | 文档结构完整，逻辑清晰 |
| 验收标准 | 是否定义明确的验收标准 | 列出可量化的检查项 ≥ 3条 |
| 素材准备 | 是否准备必要的素材/资源 | 素材清单完整，来源明确 |
| 文档规范 | 是否符合团队文档规范 | 格式统一，命名规范 |

**输出物**：
- 设计文档（Markdown，500-1000字）
- 验收标准清单
- 素材资源包
- 如有UI：Figma链接/截图

---

### 2.2 青狐实现阶段 🦊

| 检查项 | 验收标准 | 通过标准 |
|--------|----------|----------|
| 代码实现 | 是否按设计文档完整实现 | 功能点覆盖率 100% |
| 代码质量 | 是否符合团队编码规范 | lint检查通过，无警告 |
| 单元测试 | 是否编写单元测试 | 核心函数覆盖率 ≥ 80% |
| 集成测试 | 是否通过集成测试 | 端到端流程跑通 |
| 文档更新 | 是否同步更新相关文档 | API文档/README同步更新 |
| 性能检查 | 是否满足性能基线 | 响应时间 ≤ 基线 120% |

**输出物**：
- 源代码（符合git分支规范）
- 单元测试文件
- 集成测试报告
- 部署文档（如需要）

**禁止事项**：
- ❌ 禁止自行执行 git add/commit/push

---

### 2.3 黑狐审核阶段 🦊

| 检查项 | 验收标准 | 通过标准 |
|--------|----------|----------|
| 代码审核 | 是否完整审查代码改动 | 逐文件Review，无遗漏 |
| 规范检查 | 是否符合团队规范 | 命名/注释/结构合规 |
| 逻辑审核 | 业务逻辑是否正确 | 边界条件/异常处理覆盖 |
| 安全审核 | 是否存在安全隐患 | 无SQL注入/XSS/越权等风险 |
| 测试审核 | 测试是否充分有效 | 测试用例覆盖核心场景 |
| 文档审核 | 文档是否同步更新 | 代码与文档一致性检查 |

**审核动作**：
- ✅ **通过**：调用 `POST /tasks/<id>/review`，发消息通知执行人+花火
- ❌ **打回**：调用 `POST /tasks/<id>/reject`，必须附带详细打回原因和修复指引

**输出物**：
- 审核报告（通过/打回原因）
- 质量评级（A/B/C/D）
- 代码合并（审核通过后执行 git commit/push）

---

### 2.4 花火确认阶段 🔥

| 检查项 | 验收标准 | 通过标准 |
|--------|----------|----------|
| 需求对齐 | 是否满足原始需求 | 与派单需求100%对齐 |
| 质量达标 | 是否达到发布标准 | 黑狐审核通过，无遗留问题 |
| 文档完整 | 是否具备完整文档 | 用户文档/运维文档齐全 |
| 验收确认 | 是否完成最终验收 | 调用 `POST /tasks/<id>/review` 标记完成 |

**确认动作**：
- ✅ **验收通过**：任务状态变为 DONE，进入归档流程
- 📦 **归档**：调用 `POST /tasks/<id>/archive`，写入 phase_label

**输出物**：
- 验收确认书
- 归档记录
- 下一Phase规划初稿

## 三、验收链使用指南

### 3.1 任务创建时

花火创建任务时，在任务描述中引用本模板：

```markdown
## 验收标准
按 [验收链模板](./验收链模板.md) 执行：
- 白狐设计阶段：输出设计文档 + 原型
- 青狐实现阶段：代码实现 + 单元测试
- 黑狐审核阶段：代码审核 + 质量评级
- 花火确认阶段：最终验收 + 归档
```

### 3.2 任务执行时

执行者按环节自检，对照验收标准清单逐项检查：
- 全部通过 → 提交审核
- 部分未通过 → 继续完善，不提交

### 3.3 审核时

审核者按环节验收标准逐项检查，给出明确结论：
- 全部通过 → 审核通过
- 任一未通过 → 打回，指明具体问题和修复方向
