# FoxBoard 审核规范

> 版本：v2.0 | 更新日期：2026-03-22 | 维护人：黑狐
> 适用范围：所有 FoxBoard 项目任务的代码审核与文档审核

---

## 一、通用审核检查项（所有任务适用）

### 1.1 代码审核

| 检查项 | 标准 | 严重程度 |
|--------|------|----------|
| Build 检查 | `npm run build` 或 `python -m py_compile` 无错误 | 阻断 |
| 测试覆盖 | pytest 全部通过，核心路径覆盖率 ≥ 80% | 高 |
| SQL有效性 | 无裸SQL注入，参数化查询 | 阻断 |
| XSS/越权 | 无前端未转义输出，无未授权访问 | 阻断 |
| 异常处理 | 关键路径有 try/except 或错误边界 | 中 |
| 命名规范 | 函数/变量名符合语言惯例（snake_case/camelCase） | 低 |
| 注释规范 | 复杂逻辑有注释，公共API有docstring | 低 |

### 1.2 文档审核

| 检查项 | 标准 | 严重程度 |
|--------|------|----------|
| 格式规范 | Markdown格式正确，无渲染断链 | 高 |
| 路径准确 | 所有引用路径为绝对路径或验证有效 | 高 |
| 内容完整 | 验收标准、数据源、操作步骤齐全 | 高 |
| 字数达标 | 符合任务要求的字数范围 | 低 |

## 二、Analytics 数据分析任务审核要点

> 新增于 Phase 9。数据分析任务除通用审核外，需额外验证以下内容。

### 2.1 指标定义审核

| 检查项 | 验收标准 |
|--------|----------|
| 指标完整性 | 至少5个核心指标，每个有明确业务定义 |
| 计算公式 | 公式正确且可执行，需对照实际数据库表结构验证 |
| 数据源存在 | 所引用的表和字段在当前数据库schema中存在 |
| 单位明确 | 时间类指标注明单位（天/小时），百分比注明精度 |

### 2.2 数据源映射审核（关键）

**必须对照数据库实际 schema 验证以下内容：**

1. **task_logs 表字段验证**：
   - 确认 `task_logs` 表是否有 `status` 字段（当前schema：无）
   - 若文档引用 `task_logs.status`，需改用 `tasks.status` 或 `tasks.started_at`
   - 若需追踪状态变更历史，需确认是否有 `task_updated` 事件携带 status 变更

2. **Cycle Time 计算验证**：
   - 当前 `tasks` 表有 `started_at` 字段可用于计算开始时间
   - 结束时间用 `updated_at` 或 `completed_at`
   - 若需精确到状态变更瞬间，需扩展 `task_logs` 事件类型

3. **Pass Rate 计算验证**：
   - 当前 event_type 包含 `task_review_failed`，可用于统计拒绝次数
   - 需确认 `task_review_passed` 事件存在以区分一次通过

4. **数据架构问题上报**：
   - 若发现文档引用的字段/表在当前schema不存在，**不阻断审核**
   - 需在审核意见中明确标注"数据层待补充"，并通知花火协调数据层任务

### 2.3 Analytics 审核 checklist

```
□ 指标定义清晰（业务含义 + 计算公式）
□ 计算公式可执行（对照实际DB schema验证）
□ 数据源映射正确（表名、字段名存在）
□ 单位明确（天/小时/百分比）
□ 可视化建议合理（图表类型与指标特性匹配）
□ 字数在要求范围内
□ 无错别字和格式错误
□ 数据层待补充项已标注（如有）
```

## 三、验收链审核要点

> 验收链：白狐设计 → 青狐实现 → 黑狐审核 → 花火确认

### 3.1 各环节审核权重

| 环节 | 审核重点 | 通过条件 |
|------|----------|----------|
| 白狐设计 | 需求理解、产出完整性、验收标准可量化 | 设计文档结构完整，验收标准 ≥ 3条可量化项 |
| 青狐实现 | 代码正确性、测试覆盖、规范遵守 | pytest通过，lint通过，功能覆盖率100% |
| 黑狐审核 | 代码质量、数据架构（analytics）、文档一致性 | 无阻断级问题，测试充分 |
| 花火确认 | 需求对齐、最终质量 | 审核通过，无遗留问题 |

### 3.2 验收链流转审核

**提交审核前，执行者须确认：**
- [ ] 代码已通过 lint/build 检查
- [ ] 单元测试已编写并通过
- [ ] 相关文档已同步更新
- [ ] 无 TODO/FIXME/XXX 遗留（审核时抽检）

**审核打回时，审核者须提供：**
- 具体问题位置（文件:行号或文档章节）
- 问题原因（为什么不对）
- 修复指引（怎么改）

## 四、Git Commit 规范（黑狐专属）

> 补充于 Phase 9。Git 操作仅限黑狐执行。

### 4.1 Commit Message 格式

```
<type>(<scope>): <描述> #TASK-ID

# 示例
feat(tasks): 新增任务归档接口 #FB-057
fix(kanban): 修复BLOCKED列不显示问题 #FB-060
docs(review): 更新审核规范纳入analytics要点 #FB-065
refactor(analytics): 重构指标计算逻辑 #FB-XXX
```

### 4.2 Type 分类

| type | 适用场景 |
|------|----------|
| `feat` | 新功能、接口、页面 |
| `fix` | Bug修复 |
| `docs` | 文档新增/更新 |
| `refactor` | 重构（不改功能） |
| `test` | 测试代码新增/修复 |
| `chore` | 构建脚本、CI配置、依赖更新 |
| `perf` | 性能优化 |

### 4.3 Commit 操作规范

1. **审核通过后才 commit**，不提前合并
2. **一次提交完整功能**，不拆分过细
3. **先 add 再 commit 再 push**，不跳过任何步骤
4. **推送后验证**：`git log --oneline -3` 确认无误

### 4.4 分支规范

- 功能分支：`feature/FB-{id}-{简述}`
- 热修复：`hotfix/FB-{id}-{简述}`
- 主分支（保护）：`main`，禁止直接推送
